# Клиентски портал (AI организация) — План за подобрения (V3)

> **Какво е този документ.** Описание-само продуктов и архитектурен brief за ревизия на клиентския
> портал (`clients.flowai.local.com`, областта `/org`). Не е имплементационен план по файлове — Claude
> Code прави конкретния план за изпълнение по него.
>
> **Произход.** V3 = вътрешният V1 + ревюто на Codex (V2), слети и **проверени спрямо реалния код**.
> Където V2 посочваше име на статус/колона/метод, е валидирано; грешните допускания са поправени и
> събрани в Приложение A („Реалност в кода"). Claude Code да се доверява на Приложение A за конкретните
> имена.
>
> **Посока.** Клиентът никога не трябва да се чуди какво става. Всяка предложена задача има реален draft
> flow, разбираемо обяснение от служителя, стъпки, инструменти, цена/модели, решение (одобри/редактирай/
> откажи) и следа в историята. Усещането е **професионален пулт за управление на AI екип**, не игра.
>
> **Repo правила (от `CLAUDE.md`).** Без тестове и без пускане на тестове. Без legacy/back-compat — старият
> път се трие, БД се нулира при нужда.

## 0. Заключени решения

1. **Запазваме визуалния слой, сменяме ДУМИТЕ.** Аватари, звезди ★, дървото на уменията, персони,
   характер-статове, цветове — **остават визуално**. Всички игрови *думи и етикети* стават професионални
   (вкл. видимия етикет „Skill Tree" → **„Карта на уменията"**, и прозата в `PRODUCT.md`/`DESIGN.md`,
   която още продава продукта като „игра"). ★ остават като визуал, но се четат като **„ниво"**.
   *(Това е леко разширение спрямо по-ранното „Skill Tree пази името" — обосновката: „Skill Tree" е
   игрова ДУМА, а водещото правило е „никога игрови термини". Визуалното дърво и звездите остават.)*

2. **Всяко предложение има flow още при предлагането — като `draft`.** Текстът на идеята („защо" + „как")
   се **извлича от агентите на flow-а** и се преразказва от 1-во лице с персоната на служителя. Одобрение →
   flow `active`; отказ → flow `inactive` + задължителна причина → памет на служителя. Край на
   „генерирай и активирай изненадващо".

3. **Цветовете кодират ФУНКЦИЯ/домейн** (маркетинг, данни, обслужване…), стабилно и централно — не по
   `id % 7`.

4. **Прозрачност: пълна техническа, видима по подразбиране.** За всяка стъпка клиентът вижда служител,
   модел/провайдър, токени, цена (USD + кредити), QA, ретраи, planner фази. Само суровите prompt-ове/
   изходи стоят в свиваем „технически" панел — не защото крием цена, а защото са шумни. Числата остават
   моноширинни и подредени (`DESIGN.md`).

**Какво НЕ пипаме:** двигателя (`FlowPlannerService`, `AgentGeneratorService`, `GraphFlowExecutor`,
`NodeExecutorService`, knowledge base, памет); БД-схемата на game-думите (няма такива в колони); звезди/
персони/аватари като визуал.

## 5. Задачи

Клиентският изглед към `AssistantTask` + `Flow`. Основен работен екран.

**5.1 Три таба (= lifecycle лещи, не глобална навигация):**
1. **За изпълнение** — одобрени; task `ready`, flow `active`; бутон „Изпълни" при manual trigger.
2. **Предложени** — flow съществува като `draft`; показва „Защо" + „Как" + инструменти + ориентировъчна
   цена + линк към draft flow; действия: Прегледай · Редактирай · Одобри · Отхвърли.
3. **Изпълнени** — поне едно завършено изпълнение; последен резултат/цена/QA + история.
Плюс филтър „Отхвърлени/Неактивни" (не главен таб) и филтри по служител/директор/статус/trigger/функция.

**5.2 Карта на задача (видимо без клик):** заглавие · кратко описание · служител (аватар, две имена, роля) ·
директор/отдел · статус (иконка+текст+цвят) · линк към flow + неговия статус (draft/active/inactive) ·
trigger · инструменти от агентите · ниво (★) · последен run (статус/дата/QA/цена/резултат); ако е предложена —
„защо я предлагам" + първите 2–3 стъпки.

**5.3 Lifecycle (споделен от UI/backend/cron).** Текущи статуси на `AssistantTask`: `proposed`,
`generating`, `ready`, `failed`, `disabled` (виж Приложение A). **Нови:** `pending_approval`, `rejected`.
Flow статуси днес: само `draft`/`active`; **нов:** `inactive`.

| Task status | Flow status | Значение |
|---|---|---|
| `proposed` | `draft` (или още няма) | Идеята е създадена; flow чака/се генерира |
| `generating` | `draft` | Flow-ът се проектира |
| `pending_approval` | `draft` | Готов за преглед, неактивен |
| `ready` | `active` | Одобрена, готова за изпълнение |
| (`running` се извежда от `FlowRun`, не е task статус) | `active` | Активно изпълнение |
| `rejected` | `inactive` | Отказано предложение (+ причина) |
| `failed` | `draft`/`inactive` | Генериране/изпълнение се провали |
| `disabled` | `inactive` | Ръчно изключена |

**5.4 Flow статусът трябва да значи нещо (run-safety).** Днес run-ът се пуска само на база активна версия +
активни nodes, **без** проверка на `flow.status` (виж Приложение A). Добави гейт на двете места:
`FlowRunController::run` и `TaskRunService::launchReadyRun` отказват flow, който не е `active`. Подравни с
вече съществуващия `Flow::scopeRunnable()` (който филтрира `is_archived`) — да изисква и `status='active'`.
Клиентските списъци не показват `draft`/`inactive` като изпълними; линкът към draft builder е
преглед/редакция, не run. `inactive` е каноничният отказан/изключен статус (без втори термин като `paused`).

## 6. Предложение + draft flow (ЯДРОТО)

**6.1 Source of truth.** За task-предложения: `AssistantTask` (задача/собственик/статус) + `Flow` (реалния
pipeline) + `FlowVersion`/`FlowNode`/`FlowEdge` (стъпките). `OrgProposal` остава **само** за структурни
предложения (наемане/освобождаване/мандат/орг-промяна). Task-предложение НЕ минава първо като generic
`OrgProposal`, който чак при одобрение прави задача — това противоречи на „предложението вече има flow".

**6.2 Метаданни на предложението.** `assistant_tasks` няма json колона за бриф (виж Приложение A) —
**добави** напр. `proposal` (json), за да не се извлича всеки път от prompt-ове. Примерна форма:
```json
{ "rationale": "защо сега", "expected_impact": "какво подобрява",
  "steps": [{"order":1,"node_key":"...","node_name":"...","summary":"Ще...","tools":["web_search"]}],
  "tools": ["..."], "estimated_cost": {"credits":0,"usd":0},
  "generated_from_flow_version_id": 1, "brief_generated_at": "" }
```
Плюс полета за решение: `approved_at`, `approved_by`, `rejected_at`, `rejected_by`, `rejection_reason`.

**6.3 Генериране на обяснението.** След като flow-ът е записан (draft): прочети версията/nodes/edges/tools/
prompts → подреди по DAG вълни → построй кратък „implementation brief" → преразкажи го от името на служителя
(тон/роля) → запиши в `assistant_tasks.proposal`. Реализирай като нов сервиз, напр. `TaskProposalBriefService`.
Вход: `AssistantTask`, `OrgMember`+`Persona`, `Flow`/`FlowVersion`, node prompts/tools/models, KB facts/
summary, `MemberMemoryService::reflectionBlock($member)`, поуки от откази на същия служител. Изход: rationale,
expected impact, steps, tools, ориентировъчна цена, и warning ако flow-ът е твърде общ/без инструменти.

**6.4 Пример (вместо „Седмичен пост"):**
> Предлагам LinkedIn пост по темата, която най-добре пасва на текущите услуги и аудиторията на фирмата —
> за последователно присъствие, без общ текст без ясна цел.

Стъпки (извлечени от агентите, с гласа на служителя): 1) ще прегледам наличната информация за фирмата и ще
избера конкретна тема; 2) ще потърся актуални външни сигнали при нужда; 3) ще формулирам кука от 1–2
изречения; 4) ще структурирам поста като въведение → основна част → ясен CTA; 5) ще предложа платформа, ден
и час; 6) финална проверка за тон, яснота и да не повтаря предишни публикации.

**6.5 Одобрение.** `Flow: draft → active`; `task: pending_approval → ready`; запиши `approved_at/by`; пиши
`org_event` (`task_approved`); задачата отива в „За изпълнение". Опция „Одобри и пусни" → стартира run след
активиране.

**6.6 Редакция преди одобрение.** Минимум: заглавие, описание, brief copy, trigger/график, ниво (model-cost),
инструменти на intent-ниво ако е безопасен реплан. По-дълбоко: линк към builder-а на draft flow-а; след
промяна в графа brief-ът се регенерира или се маркира „остарял" с ясно UI предупреждение.

**6.7 Отказ.** UI изисква кратка причина; `Flow: draft → inactive`; `task: pending_approval → rejected`;
запиши `rejection_reason`; пиши `org_event` (`task_rejected`); причината → памет на служителя (6.8).

**6.8 Памет от откази (внимание към реалния код).** `MemberMemoryService` е **само за четене** над
`flow_memories` (`lessons()`, `runStats()`, `reflectionBlock()`) — няма метод за запис, няма embeddings (виж
Приложение A). Затова добави **път за запис**: запиши поуката от отказа като `flow_memory` ред (напр.
`kind='lesson'`, вързан за `flow_id` на draft flow-а — така членовият `flowIds()` я хваща и `reflectionBlock()`
я инжектира автоматично) **или** добави нов store-метод + `rejectionLessonsBlock($member)`. Гарантирай, че
`TaskProposalBriefService` инжектира тези поуки при следващото предложение; лимитирай броя, за да не раздува
промпта; не цитирай дословно отказите в UI без нужда.

**6.9 Генерирането НЕ активира автоматично (корекция на текущия път).** Днес `TaskRunService::dispatchGeneration`
създава flow `draft`, после `GenerateAgentsCommand` го прави `active` и `finishAssistantTask` слага задачата
`ready` + вика `autoRunAfterGenerate`. Ново: за org task-предложение **махни насилственото активиране** —
след persist задачата става `pending_approval` (не `ready`), flow-ът остава `draft`, `autoRunAfterGenerate`
не се вика. Auto-run е допустим само за вече одобрена `ready` задача с `active` flow. `DirectorAgentService::tick()`
не пуска `proposed`/`pending_approval` задачи.

## 7. Решения (Decision Box)

Inbox за всичко, което чака човек.

**7.1 Източници.** `DecisionBoxService::pending()` днес агрегира само `OrgProposal(pending)` +
`FlowRun(waiting_approval)` (виж Приложение A). **Добави трети източник:** `AssistantTask(pending_approval)`
(предложени задачи с draft flow).

**7.2 Карта за предложена задача:** предложено от (служител: аватар, две имена, роля) · заглавие · rationale ·
expected impact · стъпки от draft flow-а · инструменти · ниво · ориентировъчни кредити/USD · линк към draft
builder · линк към профила; действия: Одобри · Одобри и пусни · Редактирай · Отхвърли.

**7.3 Отказ с причина (не silent бутон):** клик „Отхвърли" → inline/modal поле „Защо не одобряваме?"
(задължително) → submit → задачата отива в „Отхвърлени" → причината влиза в паметта на служителя.

**7.4 Структурни предложения** (`hire`/`fire`/`mandate`): остават в `OrgProposal`; показвай достатъчно
детайл (кого засяга, защо, какво се променя) + коментар при отказ + смислен `org_event`; ако
`base_org_version_id` е остарял → ясно показвай `superseded`.

## 8. Генериране и статуси

**8.1 Единен status contract** за всички дълги client операции (примерна форма):
```json
{ "ok": true, "status": "generating", "stage": "Проектирам flow-а", "stage_key": "design",
  "percent": 45, "message": "Служителят проектира стъпките.", "next_action": null,
  "poll_url": ".../gen-status/{token}", "error": null }
```
Етапите **вече се пращат** в кеша (Подготовка → Анализ → Генериране → Проверка → Финализиране → Готово), но
front-end poll-ърът чете само `task_status` и игнорира `stage` — **покажи `stage`** навсякъде (член, задачи,
Табло, Нова задача). Никакво мълчаливо „pending".

**8.2 Етапи за task-предложение:** `queued` → `context` (фирма+служител) → `intent` → `design` → `critique` →
`persist` (draft flow) → `brief` (обяснение+стъпки) → `pending_approval`.

**8.3 Надеждност през Horizon.** `AgentGenerationLauncher` днес стартира detached OS процес с `exec` (извън
Horizon наблюдаемост, риск от стар код/конфиг). Премести към Horizon-управляван job (`GenerateAgentsJob` или
екв.); Horizon е единственият worker entrypoint; polling contract остава стабилен за UI; heartbeat/failed
състояния по правилата от repo-то.

**8.4 Достъпност на статусите:** `aria-live="polite"`/`role="status"` за нерешаващи промени; не местѝ фокуса
при всяка промяна; alert pattern за грешки, изискващи действие; progressbar с `aria-valuenow/min/max`;
reduced-motion се уважава.

## 9. Служители

**9.1 Две имена.** Имената днес са едно поле (`personas.name`, `org_members.display_name`), а роля/длъжност
**вече живее отделно** на `directors.title`/`assistants.title` (виж Приложение A). Проблемът е, че понякога в
полето за име попада роля. Изисквания: всеки Управител/Директор/Асистент има реално човешко **двусловно име**
(собствено + фамилно), генерирано/гарантирано така — **никога роля вместо име**; показвай „Име Фамилия" +
отделно ролята от `directors/assistants.title`. По избор: раздели на `first_name`/`last_name` в `personas`.
Пример: Име „Калоян Петров", Роля „Асистент съдържание". Не: Име „Маркетинг асистент" или само „Калоян".

**9.2 Профил (работен, не скрита форма).** `member.blade.php` показва видимо, без „Редактирай": аватар · две
имена · роля/отдел · био · тон/стил · умения (с функционален цвят) · ниво (★) · задачи по статус (с линк към
flow) · брой завършени · среден QA · последен run · кога следва предложение/изпълнение · какво му липсва
(9.4) · последни поуки/ограничения (ако са безопасни). „Редактирай профила" остава отделно действие с
mutable полетата.

**9.3 Действия (ясни имена):** Чат · Създай предложение (вместо неясното „Генерирай") · Нова задача за този
служител · Промени ниво · Регенерирай аватар; директор: Прегледай отдела; асистент: Виж задачите/резултатите.

**9.4 „Какво му липсва"** — от `knowledge_gaps` (+ липсващи полета в business profile, повтарящи се поуки от
откази, причини за нисък QA), но като човешки текст, не сурови records:
> За по-добри предложения ми трябва: актуални цени на услугите, целева аудитория за LinkedIn, предпочитан тон.

## 10. Цветове и Карта на уменията

**10.1 Цвят = функция.** Махни `member.id % 7`. Централно стабилно mapping (напр. `config/organization.php`):
маркетинг/съдържание→`char-coral`, продажби/клиенти→`char-blue`, данни/анализ→`char-teal`, операции→`char-amber`,
поддръжка→`char-green`, финанси/админ→`char-purple`, творчески/брандинг→`char-pink`. Един домейн = един цвят
навсякъде; цветът никога не е единственият сигнал.

**10.2 Умения ≠ задачи.** Днес `skill-tree.blade.php` показва задачите на асистента като „умения" — объркващо.
Раздели: **Умения** = стабилни компетентности (проучване, писане на постове, анализ на ревюта, работа с Google
Places, извличане от сайт); **Задачи** = конкретните flows. Картата на уменията показва: Управител → Директори
→ Служители; функционален домейн; умения; брой задачи; активни/завършени/неуспешни runs; ниво; отворени
knowledge gaps.

**10.3 Легенда:** цвят = функция; ★ = ниво (model-cost мощност/цена); статус-точка = текущо изпълнение; badge
„липсва информация" = knowledge gap; badge „чака решение" = pending предложение.

## 11. Хроника

История на важните събития, не лог на всеки tick.

**11.1 Влиза:** назначен/освободен служител; променен мандат; предложена/одобрена/отхвърлена(+причина) задача;
активиран flow; завършено изпълнение (кратък резултат+QA+цена); неуспешно изпълнение (+причина); добавено/
обновено важно знание; дневен директорски/управителски digest само при реални промени.

**11.2 НЕ влиза:** no-op tick; текст почти като предишния; липса на ново предложение/run/знание/структурна
промяна; вътрешен polling/status.

**11.3 Dedup и no-op guard.** Източникът на спама: `DirectorAgentService::tick()` пише `review` събитие на
**всеки** тик, а scheduler-ът пуска тик на всеки директор всеки час. Добави no-op guard в `tick()` и
`OrgReviewService::review()` (нищо ново → нищо не се пише); signature/hash срещу повтарящи се резюмета;
групиране по ден; филтри по тип/служител; drawer с връзки към task/flow/run.

**11.4 Таксономия.** Текущите `org_event` типове са общи (`hire/fire/reassign/mandate_change/approval/action/
review`). Добави по-точни (или поне `meta.subtype`): `task_proposed`, `task_approved`, `task_rejected`,
`task_completed`, `task_failed`, `flow_activated`, `knowledge_added`, `daily_digest`.

## 12. Run viewer и техническа прозрачност

Клиентът вижда не само резултата, а как системата е стигнала до него. (Решение 4: техниката е видима по
подразбиране; само суровите prompt-ове/изходи са в свиваем drawer.)

**12.1 Детайл на изпълнение** (втори слой към `client/runs/result.blade.php`): flow · задача · служител ·
started/completed · обща цена USD · общи кредити · общо токени · QA summary · брой опити · финален изход ·
node timeline.

**12.2 Node timeline (реални колони — виж Приложение A):** име · роля · `model_used` · статус · `duration_ms` ·
`prompt_tokens`/`completion_tokens`/`tokens_used` · `cost_usd` · `qa_score` · брой опити (= брой `node_runs`
реда за този `node_key`; текущият = последният по `id`) · инструменти · входни predecessor-и (namespaced
summary) · `output`. Суровият `raw_output`/prompt — в свиваем drawer.

**12.3 Planner прозрачност.** `agent_generation_logs` **няма `phase` колона** — редовете на едно генериране
се групират по `token` (виж Приложение A). Покажи per ред: `provider`, `model`, `cost_usd`, `duration_ms`,
`status`, `error`; линк към builder графа. За клиента това е секция „Технически детайли".

## 13. Нова задача

Не бива да създава усещане, че нещо изчезва в pending AJAX.
Поток: клиентът описва задача → автоматично рутиране или избран служител → UI веднага показва служителя →
генерирането показва stage-by-stage прогрес → при готов draft flow се показва proposal карта → клиентът
избира Одобри / Одобри и пусни / Редактирай / Отхвърли. Дори когато клиентът сам е създал задачата,
прегледът е полезен (flow-ът може да е зле проектиран); „Одобри и пусни" пази скоростта.
Copy: „Възложи задачата" → „Създай предложение"; „Асистентът проектира flow-а по своята персона" →
„Служителят проектира работния flow"; „Готов за изпълнение" → „Готово за преглед"; „Изпълни сега" се показва
чак след активиране (или като „Одобри и пусни").

## 14. Заземяване на предложенията (край на „Седмичен пост")

Проблемът: seed заглавие/описание е твърде общо и става краен продукт (`OrgBlueprintSeeder` „Седмичен пост",
описание = заглавие). Правила: 1) seed блупринтите са само вътрешна насока, не крайно заглавие; 2) всяко
proposal заглавие/съдържание се генерира от контекста (фирма + служител + KB + конкретния flow); 3) при
недостатъчно информация служителят НЕ измисля уверена задача — предлага „нужна информация" или ограничена
задача за събиране на контекст; 4) brief-ът цитира кои данни ползва (KB facts, business profile, предишни
runs, knowledge gaps, поуки от откази); 5) ако flow-ът е generic/без tools/steps → proposal с warning
(„needs_review"), не преструвка за добро предложение.

## 15. Сигурност, разход и rate control

Понеже всяко предложение вече строи реален flow (= разход): кредитна резервация преди генериране остава;
лимит на брой draft предложения на служител/фирма; каданс за автоматични предложения; директор/управител
ревю не генерира масово flows без причина; rejected/inactive draft flows не са runnable и могат да се
архивират (`is_archived`)/скрият от основните списъци; всяко предложение показва **ориентировъчна** цена
преди одобрение (евристика: ниво × брой nodes × база), а след run — реалната цена.

## 18. Acceptance критерии (ръчна проверка, не тестове)

- **Език/навигация:** няма видими „куест/герой/персонаж" или „Skill Tree"; меню точно Табло·Задачи·Решения·
  Интеграции·Кредити·Хроника; mobile — същите.
- **Ширина:** всички `/org` страници са широки колкото хедъра; без вложени page-level `max-w-*` контейнери.
- **Предложения:** ново предложение прави draft flow; след генериране задачата е `pending_approval`, не `ready`;
  draft flow не е изпълним; одобрение → flow `active`; отказ → flow `inactive`, task `rejected`, изисква
  причина, причината влиза в паметта на служителя.
- **Задачи:** 3 таба; всяка задача показва служител/flow/статус/инструменти/цена/последен run; предложената —
  rationale + стъпки от nodes.
- **Статус:** генерирането показва етап, не само „pending"; „Текущ поток" не reload-ва; статусите са достъпни
  за screen readers.
- **Хроника:** no-op tick не пише събитие; повтарящи се review не се трупат; отхвърлена задача се вижда с причина.
- **Прозрачност:** run detail показва node timeline с model/provider, токени, цена, QA, опити; planner
  детайлите са видими.

## Приложение A. Реалност в кода (проверено) — авторитетни имена за Claude Code

Поправки на допускания от ревюто; ползвай тези имена, не предположения.

**Flow / активиране / run-safety**
- `flows.status` default `'draft'`; **единственото** активиране е `GenerateAgentsCommand.php:174`
  `...->update(['status' => 'active'])`. `finishAssistantTask()` слага task `'ready'` и вика
  `TaskRunService::autoRunAfterGenerate()` (което при `run_after_generate` вика `launchReadyRun()`).
- `FlowRunController::run()` гейтва на `activeVersion` + активни nodes — **НЕ** на `flow.status`.
  `TaskRunService::launchReadyRun()` гейтва на flow/member + wallet — **НЕ** на `flow.status`. (И двете
  трябва да добавят проверка `status==='active'`.)
- Flow статуси, реално ползвани: само `draft` и `active`. `inactive`/`paused`/`disabled`/`archived` **не са**
  flow статуси. Архивирането е отделна булева `flows.is_archived` (+ `archived_at`) през `Flow::scopeRunnable()`.

**AssistantTask**
- Колони: `org_member_id, current_director_member_id, flow_id, title, description, trigger, schedule, act_mode,
  approval_policy, star_tier, tier_stale, kpi, status, gen_token, run_after_generate`. **Няма json колона за
  бриф** (`kpi` е string; cast-ват се само двата bool). → за `proposal` json се добавя нова колона.
- Реално ползвани статуси: `proposed, generating, ready, failed, disabled` (`running` е само ред за
  подреждане в `QuestController`). → `pending_approval`, `rejected` са нови.

**Decision Box / approvals**
- `DecisionBoxService::pending()` агрегира `OrgProposal(status='pending')` + `FlowRun(status='waiting_approval')`.
  **Не включва** `AssistantTask`. → добавя се трети източник.
- Пауза за човешко одобрение: **FlowRun.status = `'waiting_approval'`** (`NodeExecutorService.php:216`); NodeRun
  паузата = `'paused'`; node ТИПЪТ = `'human_approval'`. Отказ при approval → run/node стават `'failed'` +
  context флаг `approvals[node]['status']='rejected'` (няма отделен `rejected` run статус).

**node_runs** (за run viewer)
- `model_used`, `cost_usd`, `qa_score` (0–100), `prompt_tokens`, `completion_tokens`, `tokens_used` (агрегат —
  **няма** `total_tokens`), `status` (`pending|running|completed|failed|skipped`, +`paused`), `duration_ms`,
  `started_at`, `completed_at`, `output`, `raw_output`, `error`, `quality_metrics`(json), `params_snapshot`(json).
- **Няма `retries` колона** — всеки опит е нов `node_run` ред за същия `node_key`; текущият = последният по `id`.

**agent_generation_logs** (planner прозрачност)
- **Няма `phase` колона** — редовете на едно генериране се групират по `token`. Колони: `provider, model,
  cost_usd, prompt_tokens, completion_tokens, duration_ms, status, error, system_prompt, user_message, options,
  raw_response, flow_id, company_id`.

**Имена / роли**
- `personas.name` (едно поле), `org_members.display_name` (едно поле) — **няма** `first_name`/`last_name`.
- Роля/длъжност живее на placement редовете: `directors.title` (+ `domain`, `mandate`), `assistants.title`
  (+ `mandate`). `org_members` няма `title`; нивото е `org_members.default_star_tier`.

**MemberMemoryService** (за паметта от откази)
- **Само за четене** над `flow_memories` (писани от `DistillFlowMemoryJob`). Public: `lessons($member,$limit)`,
  `runStats($member)`, `reflectionBlock($member)` (емитва блок „[ПОУКИ ОТ МИНАЛИ ИЗПЪЛНЕНИЯ]"). **Няма**
  record/remember метод и **няма** embeddings. → запис на поука от отказ е НОВА възможност (най-чисто: нов
  `flow_memory` ред `kind='lesson'` за `flow_id` на draft flow-а, който `reflectionBlock()` хваща автоматично).
