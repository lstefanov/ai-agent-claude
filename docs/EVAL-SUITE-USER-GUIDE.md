# Eval Suite — Ръководство за ползване

> Как да тестваш Flow-овете си и как да четеш резултатите.

---

## Какво прави Eval Suite?

Eval Suite ти дава отговор на въпроса: **„Колко добре работи моят Flow и струва ли си да плащам повече за по-скъп модел?"**

Вместо да пускаш Flow-а ръчно и да преценяваш на око — задаваш критерии за качество, Eval Suite пуска Flow-а автоматично на различни нива и ти показва матрица с оценки и разходи.

---

## Стъпка 1: Създай eval case (тестов сценарий)

Отвори Flow-а → таб **„Eval"** → **„+ Нов тест"**.

### Полета:

**Име** — разпознаваемо описание, напр. `Стандартна заявка`, `Промоция лято`, `Edge: празен вход`.

**Описание** — за твоя информация, не влиза в AI-то.

**Входни данни** — това е `input_data` JSON: каквото потребителят би написал в полето за вход на Flow-а при реален run.

```json
{
  "prompt": "Напиши имейл за промоция на нашите летни курсове по плуване.",
  "variables": {
    "tone": "официален",
    "max_words": 400
  }
}
```

**Тежест** — ако имаш 3 теста и искаш `Стандартна заявка` да тежи двойно повече от останалите: сложи 2.0 на нея, 1.0 на другите. При агрегиране на резултатите по-тежките тестове влияят повече.

**Критерии за качество** — добавяш ги един по един:

| Тип | Кога | Пример |
|-----|------|--------|
| `llm_judge` | За субективни неща | „Текстът е на официален тон" |
| `rule: word_count` | За дължина | „Между 300 и 600 думи" |
| `rule: contains_keyword` | За задължително съдържание | „Съдържа думата 'промоция'" |
| `rule: no_placeholder` | За незапълнени шаблони | „Няма [PLACEHOLDER] в текста" |
| `regex` | За специфичен формат | Email адрес, телефон |

**Тежест на критерия** — колко важен е спрямо другите критерии в същия тест.
Точност на фактите = 2.0, Граматика = 0.5 — типичен баланс.

> **Съвет:** Започни с 2–3 критерия, не повече. Прекалено много критерии размиват картината.

## Стъпка 2: Стартирай eval

На страницата „Eval Cases" → секция **„Стартирай Eval"**:

**Версии** — избираш кои версии на Flow-а да тестваш. По подразбиране е активната.

**Нива** — избираш model levels:
- Начни с **low, medium, high** — те покриват 95% от случаите.
- **ultra и god** добавяй само ако резултатите при high са под 75/100.

**Cases** — „Всички активни" е добре за рутинна проверка. Избери конкретни при регресионен тест.

Натискаш **▶ Стартирай** → системата пуска истински FlowRun за всяка комбинация (версия × ниво × case), изчаква го, оценява изхода с AI-judge и записва резултатите.

> ⚠️ **Времетраене:** Всеки eval run = истински FlowRun. При 2 версии × 3 нива × 2 cases = 12 реални изпълнения. При тежки flows с много агенти може да отнеме 10–20 минути.

## Стъпка 3: Четеш матрицата с резултати

`flows/{flow}/eval/results`

### Матрица версия × ниво

```
              │  low   │ medium │  high
──────────────┼────────┼────────┼────────
v3 (активна)  │ 71/0.003$ │ 84/0.018$ │ 91/0.047$
v2            │ 68/0.003$ │ 79/0.017$ │ 87/0.046$
```

**Числото вляво** = качество (0–100). **Числото вдясно** = средна цена на един run.

**Цветове:**
- 🟢 Зелено (≥85): отлично, готово за продукция
- 🟡 Жълто (65–84): приемливо, помисли дали не трябва по-скъпо ниво
- 🔴 Червено (<65): неприемливо, смени ниво или прегледай агентите

### Как да четеш скатер графиката

X ос = цена → дясно е по-скъпо.
Y ос = качество → нагоре е по-добро.

**Идеалната точка е горе-вляво** — максимално качество при минимална цена.

Ако точките за `medium` и `high` са близо по Y оста, но далеч по X — `medium` е по-добрият избор.

### Препоръка на оптимизатора

```
💡 Препоръка: Версия v3, ниво MEDIUM
   Score: 84/100 | Цена: $0.018 | Ефективност: 4666 точки/$
   vs HIGH: +7 точки качество, но +161% цена
   vs GOD:  +11 точки качество, но +1066% цена
```

Оптимизаторът изчислява `score / cost` за всяка точка и препоръчва **максималното съотношение** — ефективната граница на Парето. Следвай препоръката като стартова точка, но имай предвид:

- За промо материали, клиентски имейли → качеството е по-важно → може да искаш HIGH.
- За вътрешни резюмета, черновки → LOW/MEDIUM са напълно достатъчни.
- За юридически/финансови документи → никога под 85/100, дори да е скъпо.

## Стъпка 4: Детайлен преглед на един eval run

Кликаш върху клетка в матрицата → виждаш:

### Критерии с оценки

```
✅ Точност на информацията    92/100  (тежест 2.0)
   "Всички цени са правилно цитирани от ценоразписа."

⚠️ Тон и стил                 76/100  (тежест 1.0)
   "Един абзац е на разговорен вместо официален тон."

✅ Дължина (rule)             100/100  (тежест 0.5)
   342 думи — в рамките на 300–600 ✓
```

**Иконите:**
- ✅ ≥80/100 — критерият е изпълнен добре
- ⚠️ 60–79/100 — частично изпълнен, обърни внимание
- ❌ <60/100 — сериозен проблем

**Причините** от AI-judge са най-ценното — те казват ЗАЩО оценката е такава, не само числото.

### Финален изход

Виждаш точния текст, изходен от Flow-а за този тест. Можеш да го сравниш с очакваното.

### Node runs в рамките на eval

```
researcher    score: 88  cost: $0.004  model: gemini-2.0-flash
writer        score: 81  cost: $0.009  model: gemini-2.0-flash
qa_verifier   score: 79  cost: $0.005  model: gemini-2.0-flash
```

Ако `qa_verifier` дава ниски вътрешни оценки, но eval score е висок → или QA threshold е твърде строг, или judge-ът е по-снизходителен. Обратното (висок QA, нисък eval) → QA критериите не съвпадат с eval критериите ти — прегледай промптовете.

## Типични сценарии на ползване

### Сценарий 1: „Мога ли да сваля нивото?"

1. Текущото ниво е HIGH. Искаш да пестиш.
2. Имаш 2–3 eval cases.
3. Пускаш: v3 (активна) × {medium, high}.
4. Ако medium score е ≥80 и разликата с high е <8 точки → свалямо нивото.
5. Прилагаш: Builder → toolbar → смени ниво → Relevel → Запази.

### Сценарий 2: „Новата версия по-добра ли е?"

1. Редактирал си промптите → запазено като v4.
2. Пускаш: {v3, v4} × {medium}.
3. Ако v4 score > v3 score с ≥5 точки → активираш v4.
4. Ако разликата е <3 точки → не си струва смяната.

### Сценарий 3: „Защо качеството падна след сряда?"

1. Nightly eval (ако е конфигурирано) записва резултати всяка нощ.
2. Страницата Results → превключваш дата диапазон → виждаш score trend.
3. Сряда показва спад при `accuracy` критерия → прегледай промена в Knowledge Base или промпт, направена в сряда.

### Сценарий 4: „Трябва ли ultra за юридически документи?"

1. Създаваш eval case с критерии: Пълнота (тежест 3), Точност (тежест 3), Правна коректност (тежест 2).
2. Пускаш: v_latest × {medium, high, ultra}.
3. Ако ultra е единственото ниво с ≥90/100 на `Правна коректност` → оправдано.
4. Ако high постига 88/100 при 30% от цената → high е изборът.

## Как да правя добри eval cases

### ✅ Добри практики

- **Реален вход** — използвай заявки, близки до това, което реален потребител би написал.
- **Измерими критерии** — „Текстът е на официален тон" е измеримо. „Текстът е добър" — не е.
- **2–5 критерия** на case — не повече. Фокусирай се на най-важните.
- **Edge cases** — добави поне 1 тест с трудна заявка (празен вход, необичайна тема).
- **Тегло ≠ важност само за теб** — задавай тегло спрямо колко критичен е критерият за бизнеса.

### ❌ Грешки, которых да избягваш

- **Не задавай перфектистки критерии** — score 100/100 е практически недостижим. 85+ е отличен.
- **Не разчитай само на `llm_judge`** — добави поне 1 `rule` критерий (дължина, ключова дума) — тя е детерминистична и не зависи от LLM.
- **Не пускай eval само при проблем** — пускай редовно (или nightly), за да следиш тренда.
- **Не сравнявай eval runs от различни дати** без да знаеш дали се е променило нещо в системата.

## Как да настроиш nightly eval

Ако `flows:run-evals` е конфигурирана в Scheduler (в `routes/console.php`):

```php
Schedule::command('flows:run-evals')->dailyAt('03:00');
```

Системата автоматично:
1. Взима всички активни версии с поне 1 eval case.
2. Пуска eval на текущото ниво на версията.
3. Записва резултатите.
4. Ако score е паднал с >10 точки спрямо предишния ден → опционален alert (email/Slack, ако е конфигурирано в Delivery Settings на Flow-а).

## Речник

| Термин | Значение |
|--------|----------|
| **Eval case** | Един тестов сценарий с вход + критерии |
| **Eval run** | Едно изпълнение на eval case за конкретна версия + ниво |
| **Judge** | AI, който оценява изхода по критериите (обикновено Gemini) |
| **Score** | Претеглена средна оценка 0–100 от всички критерии |
| **Ефективност** | Score / цена — колко качество получаваш за всеки долар |
| **Матрица** | Таблица версия × ниво с score и цена |
| **Парето граница** | Точките, при которых повишаването на качество изисква непропорционален ръст на цена |
